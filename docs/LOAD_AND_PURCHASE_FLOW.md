# Нагрузка и выдача подписок (покупка ключа)

## Поведение при наплыве запросов на создание ключа

Все апдейты Telegram (в т.ч. нажатие «Купить» / создание ключа) проходят через **ConcurrencyLimiterMiddleware**:

- Одновременно обрабатывается не более **110** апдейтов (значение `CONCURRENT_UPDATES_LIMIT` = `DB_POOL_SIZE` + `DB_MAX_OVERFLOW`).
- Каждый из этих 110 держит одно соединение с БД на время обработки (в т.ч. вызов панелей Remnawave/3x-ui).
- Остальные апдейты **ждут в очереди** на семафоре.
- Если апдейт ждал в очереди **дольше 45 секунд** (`UPDATE_STALE_AGE_SEC`), он **отклоняется**: пользователь получает сообщение «Время ожидания истекло. Нажмите ещё раз.» / «Сейчас высокая нагрузка…».

Итог при 10 000 одновременных запросах на покупку:

- Бот **не падает и не зависает**: обрабатываются 110 запросов, остальные стоят в очереди.
- Часть пользователей (те, кто успел обработаться за ~45 с) получают ключ.
- Большая часть получит отказ по таймауту и должна нажать ещё раз позже.

Дополнительные ограничения внутри создания ключа:

- **Remnawave**: не более 20 одновременных запросов к API (`REMNAWAVE_MAX_CONCURRENCY`).
- **3x-ui**: семафор 2 на сервер при создании клиента в сценарии выбора страны.

## Рекомендации при ожидании пиков (тысячи одновременных покупок)

1. **Увеличить время ожидания в очереди**  
   В `core/cache_config.py`: поднять `UPDATE_STALE_AGE_SEC` (например до 60–90 сек), чтобы больше запросов успели обработаться до отсечения по таймауту.

2. **Увеличить пул БД и лимит апдейтов**  
   В `config.py`: увеличить `DB_POOL_SIZE` и `DB_MAX_OVERFLOW` (и `max_connections` в PostgreSQL), чтобы одновременно обрабатывать больше покупок. Тогда же вырастет `CONCURRENT_UPDATES_LIMIT`.

3. **Горизонтальное масштабирование**  
   Несколько инстансов бота (webhook) с общими Redis и PostgreSQL: каждый инстанс обрабатывает свою долю апдейтов, лимит 110 — на каждый инстанс.

4. **Очередь покупок (при очень больших пиках)**  
   Вынести создание ключа в фоновую очередь (Redis/ Celery и т.п.): пользователь нажимает «Купить» → запрос ставится в очередь, в боте приходит «Вы в очереди, ключ придёт в чат». Воркеры создают ключи и отправляют сообщение пользователю. Так не ограничивать число «ожидающих» лимитом одновременных апдейтов.
